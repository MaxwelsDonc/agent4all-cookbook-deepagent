# Agent4all 课程设计思考

这份文档记录 Agent4all 的课程设计原则，后续所有教程内容都应该围绕这些原则展开。

## 项目定位

Agent4all 不是给高手看的速查文档，而是给小白的一条完整上手路径。

这个项目的目标是：让一个什么都不懂的人，能够跟着课程一步一步走，最终理解 DeepAgent 的核心概念，并成功搭建、运行一个自己的 Agent。

课程不应该只是罗列概念，也不应该只是堆代码。每一节都要回答三个问题：

1. 这个概念解决了什么问题？
2. 用一个简单例子怎么理解？
3. 在代码里它对应什么？

## 核心哲学

### 从小白视角出发

课程默认读者一开始不懂 Agent、不懂大模型工程，也可能不熟悉 Python、终端、API Key、环境变量等基础知识。

因此教程不能默认读者“应该知道”。只要某个知识点会影响他能否继续往下走，就需要用简单语言解释清楚。

### 先理解，再动手

不要一上来就让读者复制命令或代码。

每一步操作前都要先讲明白：

- 为什么要做这一步。
- 这一步会改变什么。
- 如果不做这一步会发生什么。
- 做完后应该看到什么结果。

### 概念必须讲清楚、讲透彻

Agent 相关概念很多，比如 model、token、prompt、tool、context、memory、subagent、permission 等。

这些概念不能只给英文名和定义。每个概念都应该用固定结构解释：

```text
一句话解释
用简单例子理解
回到 Agent 里是什么意思
代码里对应哪里
常见误解
```

### 不让读者一直读文字

课程不能变成纯文字阅读。

每一阶段都应该有明确的可操作内容，让读者能看到自己的 Agent 一点点变强：

- 先跑通最小 Agent。
- 再让 Agent 有清晰角色。
- 再让 Agent 使用工具。
- 再让 Agent 读写文件。
- 再让 Agent 规划任务。
- 最后做出一个完整项目。

## 核心类比：电脑里的人

Agent4all 会把 Agent 解释成：

> 一个住在电脑里的“人”。

它不是真的人，而是一个由大模型驱动、可以根据指令思考、使用工具、执行任务的软件助手。

这个类比足够简单，也足够覆盖 Agent 的核心概念。

## 概念映射

| Agent 概念 | 用“电脑里的人”理解 |
| --- | --- |
| Agent | 电脑里的任务执行者 |
| Model | 他的脑子 |
| Token | 他阅读和说话消耗的精力，也可以理解成使用模型的计量单位 |
| Prompt | 你对他说的话 |
| System Prompt | 你给他的身份、岗位说明和工作规则 |
| Messages | 你们之间的聊天记录 |
| Context | 他当前能看到、知道、参考到的信息 |
| Tools | 他能使用的电脑工具，比如搜索、计算器、文件读写 |
| Tool Calling | 他决定打开某个工具去完成一件事 |
| Planning | 他先列计划，再一步一步执行 |
| File System | 他能读写的文件夹和笔记本 |
| Memory | 他长期记住的偏好、经验和历史信息 |
| Subagent | 他请来的其他“电脑里的人”或专门助理 |
| Permission | 你允许他做什么、不允许他做什么 |
| Human-in-the-loop | 重要操作前，他先问你确认 |
| Sandbox | 给他练习和试错的安全环境 |
| Streaming | 他边做事边向你汇报进度 |
| LangSmith | 工作日志、复盘记录和调试面板 |
| DeepAgents | 一套帮你快速创建这个“电脑里的人”的工具包 |

## 示例使用方式

课程标题应该保持专业，不要把标题都改成比喻。

例如章节标题可以是：

```text
模型与 Token
Prompt 与 System Prompt
Tools 与 Tool Calling
Planning 任务规划
文件系统与上下文管理
Subagent 子 Agent
权限、安全与人工确认
```

但在正文解释时，要使用简单例子帮助理解。

例如解释 Tool Calling：

> 一个再聪明的人，也不能凭空知道今天的天气。
>
> 他需要打开天气 App。
>
> 对 Agent 来说，天气 App 就是工具；决定打开天气 App 这个动作，就是 tool calling。

例如解释 System Prompt：

> 你请一个人做事，如果只说“帮我写个计划”，他可能写得很随意。
>
> 如果你提前说明“你是一位耐心的新手教练，必须用简单语言，每天任务不要超过 1 小时”，他就会按这个身份和规则工作。
>
> 这就是 system prompt。

## 课程主线

课程应该形成一条清晰路径：

```text
我什么都不懂
  ↓
我知道 Agent 可以理解成电脑里的人
  ↓
我理解这个人需要脑子、指令、工具和权限
  ↓
我准备好 Python 环境
  ↓
我跑通第一个最小 Agent
  ↓
我让 Agent 有清晰角色
  ↓
我让 Agent 使用工具
  ↓
我让 Agent 读写文件
  ↓
我让 Agent 规划任务
  ↓
我理解 subagent、memory、permission
  ↓
我完成一个可运行的 DeepAgent 项目
```

## 贯穿项目

课程需要一个贯穿项目，但这个项目不应该抢走概念解释的主线。

建议使用：

> MaxLance Assistant，一个帮助小白学习 DeepAgent 的学习助理。

读者一边学习 DeepAgent，一边亲手搭建一个能够帮助自己学习 DeepAgent 的 Agent。

MaxLance 这个名字来自两个贡献者：Maxwel 和 Lancer。它可以先作为单个学习助理存在，后续在多 Agent 教学里，再扩展出 Hugging Face 助理、虚拟角色助理、文档助理等不同领域的协作角色。

这个项目可以逐步演进：

1. 第一个版本：只会回答问题。
2. 第二个版本：有明确的 system prompt，像一个耐心的新手教练。
3. 第三个版本：能调用简单工具。
4. 第四个版本：能保存学习计划和笔记。
5. 第五个版本：能先规划再执行。
6. 第六个版本：能搜索资料。
7. 最终版本：能根据用户情况制定学习路径、保存结果、生成练习、根据反馈调整计划。

## 内容写作模板

每一篇教程建议使用这个结构：

```md
# 标题

## 这一节你会学到什么

## 一句话讲清楚

## 为什么需要它

## 用一个简单例子理解

## 回到 Agent

## 代码里长什么样

## 跟着做

## 你应该看到什么结果

## 常见错误

## 小结

## 下一步
```

## 判断一篇教程是否合格

一篇教程写完后，需要检查：

- 小白能不能知道这一节为什么存在。
- 是否解释了关键概念，而不是只给代码。
- 是否有简单例子帮助理解。
- 是否有可以运行或验证的结果。
- 是否说明了常见错误。
- 是否把读者自然带到下一步。

## 最终目标

Agent4all 最终应该让读者获得两样东西：

1. 对 DeepAgent 核心概念的真实理解。
2. 一个自己亲手搭建、可以运行、可以继续改造的 Agent。

如果读者学完后只是看懂了一些术语，但没有跑出 Agent，这个课程就失败了。

如果读者虽然一开始什么都不懂，但跟着课程能一步步跑起来、理解每个部件的作用，并能解释给别人听，这个课程才算成功。
